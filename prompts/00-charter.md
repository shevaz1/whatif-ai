# Project Charter — Apps in Toss WebView App

## 1. 프로젝트 목적
이 프로젝트는 **Apps in Toss WebView 기반 미니앱**을 빠르게 기획·구현·검증하기 위한 것이다.  
사용자가 제공하는 **아이디어 / 컨셉 / 문제 정의**를 입력으로 받아,  
**기획 → 화면 → 구현 → 검증**까지 자동으로 진행한다.

---

## 2. 배포/플랫폼 전제 (절대 변경 금지)
- 플랫폼: **Apps in Toss**
- 배포 방식: **WebView**
- 프레임워크: **React + TypeScript**
- 빌드 시스템: **Granite**
- UI 시스템: **TDS 필수 사용 (비게임 WebView 앱)**
- 디자인: **Toss Look 가이드 준수** — `desigin/toss-look.md`와 `src/design/tokens.ts`를 따른다. 간격·모서리·타이포는 토큰만 사용하고, 임의 숫자(7, 10, 13px 등) 하드코딩 금지.

Granite 빌드 결과물은 반드시 `granite.config.ts`의 `outdir`와 일치해야 한다.
Apps in Toss 정책/SDK/제약은 구현 전에 반드시 문서 근거를 확인한다.

---

## 3. 작업 기본 원칙
1. **문서 우선**
   - Apps in Toss SDK, WebView 제약, 검수 기준이 관련되면 `@docs`로 근거 요약 후 작업한다.
2. **작게 쪼개기**
   - 모든 작업은 30~60분 단위의 “패킷”으로 나눈다.
3. **완료조건(DoD) 선작성**
   - 코드 작성 전에 반드시 DoD를 먼저 정의한다.
4. **최소 구현**
   - MVP에 불필요한 기능/설계/리팩토링은 하지 않는다.
5. **검수 관점 선반영**
   - 운영 / 기능 / 디자인 / 보안 관점에서 위험 요소를 미리 제거한다.

---

## 4. 아이디어 입력 시 고정 처리 절차
사용자가 아이디어를 입력하면 반드시 아래 순서로 진행한다.

### Step A. 1페이지 기획 요약
- 문제 정의
- 타깃 사용자
- 핵심 가치
- 필수 기능 3개
- 후순위 기능 3개
- WebView/App-in-Toss 관점 제약

### Step B. 화면 & 흐름 설계
- 라우트 목록
- 각 화면의 상태 (Loading / Empty / Error / Success)
- 주요 사용자 플로우

### Step C. 구현 계획
- 패킷 단위 작업 목록
- 각 패킷의 DoD (5개)
- 의존 문서/SDK/설정

### Step D. 구현
- DoD를 만족하는 최소 변경만 수행
- TDS 우선 사용
- Granite outdir 규칙 유지

### Step E. 결과 보고
- 변경 요약
- DoD 체크
- 검증 방법
- 리스크/다음 단계

---

## 5. 결과물 출력 형식 (항상 지킬 것)
- 요약 (3줄)
- 완료조건 체크리스트
- 변경 파일 목록
- 검증 방법
- 리스크 2개 + 다음 패킷 제안

---

## 6. 금지 사항
- Apps in Toss / WebView / TDS / Granite 규칙 무시
- 문서 근거 없는 SDK 사용
- 한 번에 큰 기능 추가
- MVP 단계에서 과도한 추상화
